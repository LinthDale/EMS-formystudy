## 1. 系統總覽

### 1.1 容器與端口

| 容器 | 端口 | 用途 |
|------|------|------|
| `ems-simulator` | 5020 (Modbus), 8001 (REST) | 模擬電表 + 控制 API |
| `ems-gateway` | — | Modbus → MQTT |
| `ems-mosquitto` | 1883 | MQTT Broker |
| `ems-ingest` | — | MQTT → 資料庫 |
| `ems-timescaledb` | 5432 | 時序資料庫 |
| `ems-query` | 3001 | REST API (查詢歷史) |
| `ems-grafana` | 3000 | 儀表板 + 告警 |
| `ems-kc-modbus-sim` | 5021 (Modbus) | KC 工廠 PLC 模擬器 |
| `ems-kc-mqtt-sim` | — | KC JSON 溫濕度感測器 |
| `ems-kc-gateway` | — | KC 工廠 Modbus → MQTT |
| `ems-kc-ingest` | — | KC 工廠 MQTT → TimescaleDB |
| `ems-kc-mcp-server` | 127.0.0.1:8765 | MCP 讀寫設備介面 |

### 1.2 資料流

```
電表資料流：
Simulator → Gateway → Mosquitto → Ingest → TimescaleDB → Grafana → Telegram
 (5020)     (Modbus)    (MQTT)     (SQL)     (儲存)      (告警)    (通知)

KC 工廠資料流：
KC Modbus Sim → KC Gateway → Mosquitto → KC Ingest → TimescaleDB → Grafana
    (5021)       (Modbus)      (MQTT)       (SQL)      factory_measurements

KC JSON Sensor → Mosquitto → KC Ingest → TimescaleDB → Grafana
                 factory/sensor/temp_01     factory_measurements

MCP 控制流：
MCP Client → kc-mcp-server → simulator / kc-modbus-sim
             127.0.0.1:8765
```

## 2. 啟動與停止

### 2.1 啟動全部服務

```bash
cd ~/synaiq/EMS
docker compose up -d
```

這會同時啟動既有電表鏈路與 KC 工廠/MCP 服務。若只想重建並啟動 KC 相關服務：

```bash
docker compose build kc-modbus-sim kc-mqtt-sim
docker compose up -d kc-modbus-sim kc-mqtt-sim kc-gateway kc-ingest kc-mcp-server
```

### 2.2 停止服務（保留資料）

```bash
docker compose down
```

```bash
docker compose down -v
docker compose up -d
```

### 2.4 檢查服務狀態

```bash
docker compose ps
```

## 3. 告警機制操作

### 3.1 告警觸發條件

目前設定的告警規則分為兩個群組（business vs pipeline，職責分離見 §3.6）：

**Group 1：業務告警（EMS-功率告警）**

| 規則名稱 | UID | 條件 | 持續時間 | 通知方式 |
|---------|-----|------|---------|---------|
| 功率超限 | `ems-power-kw-high` | power_kw 平均 > 100 kW | 30 秒 | Telegram |

**Group 2：管線健康告警（EMS-管線健康）**

| 規則名稱 | UID | 條件 | 持續時間 | 通知方式 |
|---------|-----|------|---------|---------|
| 電表資料寫入停滯 | `ems-pipeline-electricity-stale` | 過去 2 分鐘 `electricity_measurements` 0 筆 | 1 分鐘 | Telegram (critical) |
| 工廠資料寫入停滯 | `ems-pipeline-factory-stale` | 過去 2 分鐘 `factory_measurements` 0 筆 | 1 分鐘 | Telegram (critical) |

### 3.2 手動觸發告警（測試用）

**方法 A：調高模擬功率（測試業務告警）**

```bash
# 將基礎電流調高到 200A，功率會超過 100kW
curl -X POST "http://localhost:8001/config?current_base_a=200"

# 等待 30-90 秒後，Telegram 會收到告警
# 查看告警狀態
curl -s -u admin:admin http://localhost:3000/api/prometheus/grafana/api/v1/rules | jq '.data.groups[].rules[] | {name, state}'
```

**方法 B：注入故障模式**

```bash
# 凍結數值（模擬通訊斷線）
curl -X POST "http://localhost:8001/inject-fault?mode=freeze"

# 數值歸零（模擬電表故障）
curl -X POST "http://localhost:8001/inject-fault?mode=zero"

# 恢復正常
curl -X POST "http://localhost:8001/inject-fault?mode=none"
```

**方法 C：停 ingest 服務（測試管線健康告警）**

```bash
# 停電表 ingest → 2~3 分鐘後 Telegram 收到「電表資料寫入停滯」
docker compose stop ingest

# 停 KC ingest → 2~3 分鐘後 Telegram 收到「工廠資料寫入停滯」
docker compose stop kc-ingest

# 復原（1 分鐘內告警自動解除，會收到 RESOLVED 通知）
docker compose start ingest kc-ingest
```

> 管線告警的 `noDataState/execErrState` 都設為 `Alerting`（沒資料 = 真的壞了）。
> 業務告警則設為 `OK`（避免管線中斷時誤觸功率告警）。設計理由見 §3.6。

### 3.3 恢復正常狀態

```bash
# 恢復正常功率參數
curl -X POST "http://localhost:8001/config?current_base_a=100&current_swing_a=40"

# 確認故障模式已關閉
curl -X POST "http://localhost:8001/inject-fault?mode=none"
```

### 3.4 查看目前告警狀態

```bash
# 透過 Grafana API 查詢
curl -s -u admin:admin http://localhost:3000/api/prometheus/grafana/api/v1/rules \
  | jq '.data.groups[].rules[] | {name: .name, state: .state}'

# 或直接開瀏覽器
# http://localhost:3000/alerting/list
```

### 3.5 業務告警 vs 管線健康告警（設計分工）

兩組告警的 `noDataState` / `execErrState` 設定刻意不同，因為它們監控的是不同層次：

| 告警分組 | 監控對象 | noDataState | execErrState | 為什麼 |
|---------|---------|-------------|--------------|-------|
| 業務告警（功率超限） | 業務指標（kW） | **OK** | **OK** | 沒資料/SQL 錯時不該告警，否則改 schema 或停 ingest 都會誤觸 |
| 管線健康（寫入停滯） | 基礎設施（資料流） | **Alerting** | **Alerting** | 沒資料就是真的壞了 — 這正是這組告警要抓的 |

**為什麼分開**：以前業務告警吃下「沒資料」的責任，結果一改 schema 或一停 ingest 就誤觸 Telegram。把兩種職責拆開後：

- 改業務閾值（如 100 → 120 kW）只動業務告警
- 改 DB schema / 停服務除錯時，只有管線告警會出聲，且這正是預期行為
- 收到「管線停滯」通知 → 看 §1.2 資料流圖，依鏈路順序排查容器

### 3.6 手動測試 Telegram 通知

```bash
curl -u admin:admin -X POST http://localhost:3000/api/alertmanager/grafana/config/api/v1/receivers/test \
  -H 'Content-Type: application/json' \
```

## 4. API 操作指南

### 4.1 Simulator API (端口 8001)

#### 健康檢查

```bash
curl http://localhost:8001/health
# 回應: {"status":"ok"}
```

#### 查看目前模擬參數

```bash
curl http://localhost:8001/config
# 回應:
# {
#   "noise_voltage_v": 3.0,
#   "current_base_a": 100.0,
#   "current_swing_a": 40.0,
#   "noise_current_a": 2.0,
#   "power_factor": 0.85,
#   "period_seconds": 3600.0,
#   "fault_mode": "none"
# }
```

#### 修改模擬參數

```bash
# 調整基礎電流（影響功率）
curl -X POST "http://localhost:8001/config?current_base_a=150"

# 調整功率因數
curl -X POST "http://localhost:8001/config?power_factor=0.9"

# 同時調整多個參數
curl -X POST "http://localhost:8001/config?current_base_a=120&current_swing_a=30&power_factor=0.88"
```

#### 功率計算公式

```
P (kW) = √3 × V_LL × I × cos(φ) / 1000
       = √3 × 380V × current_base_a × power_factor / 1000
```

**快速對照表：**

| current_base_a | power_factor | 預估功率 (kW) |
|---------------|--------------|--------------|
| 100 | 0.85 | ~56 kW |
| 150 | 0.85 | ~84 kW |
| 180 | 0.85 | ~101 kW (觸發告警) |
| 200 | 0.85 | ~112 kW |

#### 注入故障

```bash
# 凍結 - 數值停止更新
curl -X POST "http://localhost:8001/inject-fault?mode=freeze"

# 歸零 - 所有數值變 0
curl -X POST "http://localhost:8001/inject-fault?mode=zero"

# 恢復正常
curl -X POST "http://localhost:8001/inject-fault?mode=none"
```

### 4.2 Query API - PostgREST (端口 3001)

#### 查詢最新資料

```bash
# 電表最新 10 筆

# 電表最新 1 筆
```

```bash
curl "http://localhost:3001/factory_measurements?device_id=eq.sensor-001&order=time.desc&limit=10"
```

#### 時間範圍查詢

```bash
# 過去 1 小時
curl "http://localhost:3001/electricity_measurements?time=gte.$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%S)&order=time.desc"
```

```bash

curl "http://localhost:3001/factory_measurements?temperature=gt.28&order=time.desc&limit=10"
```


### 4.5 資料庫直接操作 (端口 5432)

#### 進入 psql

```bash
docker exec -it ems-timescaledb psql -U postgres -d ems
```

#### 常用 SQL 查詢

```sql
-- 最新 5 筆資料
SELECT time, device_id, voltage, current, power_kw 
FROM electricity_measurements 
ORDER BY time DESC 
LIMIT 5;

-- 統計資料量
SELECT COUNT(*), MIN(time), MAX(time) FROM electricity_measurements;

-- 過去 1 小時平均功率
SELECT AVG(power_kw) as avg_power 
FROM electricity_measurements 
WHERE time > NOW() - INTERVAL '1 hour';

-- 每小時最大功率
SELECT 
  time_bucket('1 hour', time) as hour,
  MAX(power_kw) as max_power
FROM electricity_measurements
WHERE time > NOW() - INTERVAL '24 hours'
GROUP BY hour
ORDER BY hour;

-- KC 工廠最新資料
SELECT time, device_id, device_type, temperature, humidity, pressure, motor_speed, pump_on, valve_open
FROM factory_measurements
ORDER BY time DESC
LIMIT 10;

-- KC 工廠最近 10 分鐘資料量
SELECT device_id, device_type, COUNT(*)
FROM factory_measurements
WHERE time > NOW() - INTERVAL '10 minutes'
GROUP BY device_id, device_type;
```

### 4.6 MCP Server 操作 (端口 8765)

`ems-kc-mcp-server` 提供 MCP streamable HTTP endpoint：

```text
http://localhost:8765/mcp
```

這不是瀏覽器網頁，也不是一般 REST API；Chrome 直接打開看到 `Not Acceptable: Client must accept text/event-stream` 是正常的。請用 MCP client 連線。

#### MCP Inspector

```bash
npx @modelcontextprotocol/inspector
```

UI 填法：

| 欄位 | 值 |
|------|----|
| Transport | Streamable HTTP |
| URL | `http://localhost:8765/mcp` |
| Command / Args / Environment | 留空 |

連上後到 `Tools`，先執行 `List Tools`，再測 `list_devices`、`read_device`。

#### curl 只驗證 header

```bash
curl -i -X POST http://localhost:8765/mcp   -H 'Content-Type: application/json'   -H 'Accept: application/json, text/event-stream'   --data '{"jsonrpc":"2.0","id":1,"method":"tools/list","params":{}}'
```

MCP device key 與 DB `device_id` 不同：

| MCP device | DB device_id | 說明 |
|------------|--------------|------|
| `energy_meter` | `sim-001` | 既有電表 simulator |
| `factory_plc` | `plc-001` | KC 工廠 PLC simulator |

安全注意：MCP server 目前沒有認證，已在 compose 中綁定 `127.0.0.1:8765`，不要直接暴露到內網或公網。

## 5. 修改警示訊息

### 5.1 修改告警規則

編輯檔案：`infra/grafana/provisioning/alerting/rules.yaml`

檔案結構（兩個 group，職責分離見 §3.5）：

```yaml
groups:
  # Group 1: 業務告警（noData/execErr → OK，避免管線中斷誤觸）
  - orgId: 1
    name: EMS-功率告警
    folder: EMS
    interval: 1m
    rules:
      - uid: ems-power-kw-high
        title: "功率超限 (power_kw > 100 kW)"
        for: 30s
        labels:
          severity: warning
        annotations:
          summary: "功率超過 100 kW"
          description: "過去 5 分鐘平均 power_kw 超過 100 kW，請確認現場狀況"
        noDataState: OK         # 沒資料 → 視為正常
        execErrState: OK        # SQL 錯 → 視為正常

  # Group 2: 管線健康告警（noData/execErr → Alerting，沒資料就是真的壞了）
  - orgId: 1
    name: EMS-管線健康
    folder: EMS
    interval: 1m
    rules:
      - uid: ems-pipeline-electricity-stale
        title: "電表資料寫入停滯 (>2 分鐘無新資料)"
        # 條件：COUNT(*) of electricity_measurements where time > now()-2m < 1
        for: 1m
        labels:
          severity: critical
          component: pipeline
        noDataState: Alerting   # 沒資料 = 壞了
        execErrState: Alerting  # SQL 錯 = 壞了
      - uid: ems-pipeline-factory-stale
        # 同上，但查 factory_measurements
```

**修改業務告警閾值：** 找到 `ems-power-kw-high` 規則內的 `params: - 100`，改成你要的數值（例如 120）

**修改管線停滯時間窗：** 找到 SQL 中的 `interval '2 minutes'`，改成你要的時間（要同時調整 `for:` 持續時間）

**修改後重新載入：**

```bash
curl -u admin:admin -X POST http://localhost:3000/api/admin/provisioning/alerting/reload
```

> 若改動的是 group 結構或 rule body（非單純標籤/註記），reload API 有時 fingerprint 偵測不到，需要 `docker compose restart grafana` 才會生效。詳見 §7.7。

### 5.2 修改通知訊息格式

編輯檔案：`infra/grafana/provisioning/alerting/contact-points.yaml`

```yaml
contactPoints:
  - orgId: 1
    name: Telegram
    receivers:
      - uid: telegram-main
        type: telegram
        settings:
          bottoken: "${TELEGRAM_BOT_TOKEN}"
          chatid: "7171144544"       # 你的 Telegram Chat ID
          message: |
            EMS 告警 [{{ .Status | toUpper }}]
            {{ range .Alerts -}}
            規則：{{ .Labels.alertname }}
            摘要：{{ .Annotations.summary }}
            說明：{{ .Annotations.description }}
```

### 5.4 新增告警規則

```yaml
      - uid: ems-voltage-low

        condition: C

            relativeTimeRange:
              from: 300
              to: 0
            datasourceUid: timescaledb-ems
            model:
              rawQuery: true
              rawSql: |
                SELECT time, voltage
                FROM electricity_measurements
                WHERE time >= now() - interval '5 minutes'
                ORDER BY time
              refId: A
          - refId: B
            datasourceUid: "-100"
            model:
              type: reduce
              expression: A
              reducer: mean
          - refId: C
            datasourceUid: "-100"
            model:
              type: threshold
              expression: B
              conditions:
                - evaluator:
                    type: lt        # lt = less than
                    params:
                      - 360
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "電壓過低警告"
          description: "過去 5 分鐘平均電壓低於 360V，可能有電力品質問題"
        notification_settings:
          receiver: Telegram
```

## 7. 故障排除

### 7.1 告警沒有觸發

1. **檢查功率是否真的超過閾值**
   ```bash
   docker exec ems-timescaledb psql -U postgres -d ems -c \
     "SELECT AVG(power_kw) FROM electricity_measurements WHERE time > NOW() - INTERVAL '5 minutes';"
   ```

2. **檢查告警規則狀態**
   ```bash
   curl -s -u admin:admin http://localhost:3000/api/prometheus/grafana/api/v1/rules | jq
   ```

3. **檢查 Grafana 日誌**
   ```bash
   docker compose logs grafana | grep -i alert
   ```

### 7.2 Telegram 收不到通知

1. **確認 Bot Token 正確**
   ```bash
   cat .env | grep TELEGRAM
   ```

2. **確認 Chat ID 正確**
   ```bash
   cat infra/grafana/provisioning/alerting/contact-points.yaml | grep chatid
   ```

3. **測試 Bot 是否能發訊息**
   ```bash
   curl -X POST "https://api.telegram.org/bot<YOUR_TOKEN>/sendMessage" \
     -d "chat_id=7171144544" \
     -d "text=測試訊息"
   ```

### 7.3 API 回應空陣列

1. **確認資料庫有資料**
   ```bash
   docker exec ems-timescaledb psql -U postgres -d ems -c \
     "SELECT COUNT(*) FROM electricity_measurements;"
   ```

2. **確認 api schema view 存在**
   ```bash
   docker exec ems-timescaledb psql -U postgres -d ems -c "\dv api.*"
   ```

3. **重啟 PostgREST**
   ```bash
   docker compose restart query
   ```

### 7.4 KC 工廠資料沒有出現在 Grafana

1. **確認 DB 有 factory 資料**
   ```bash
   docker exec ems-timescaledb psql -U postgres -d ems -c      "SELECT device_id, device_type, COUNT(*) FROM factory_measurements WHERE time > NOW() - INTERVAL '10 minutes' GROUP BY device_id, device_type;"
   ```

2. **確認 KC ingest 沒有寫入錯誤**
   ```bash
   docker compose logs --tail=80 kc-ingest
   ```

3. **重新載入 Grafana dashboard provisioning**
   ```bash
   curl -u admin:admin -X POST http://localhost:3000/api/admin/provisioning/dashboards/reload
   ```

4. **瀏覽器硬重新整理**
   ```text
   http://localhost:3000/d/ems-overview
   ```

### 7.5 MCP Inspector 連不上

1. **確認 endpoint 不是用 Chrome 直接打開**：MCP endpoint 需要 MCP client。
2. **Inspector 設定**：Transport 選 `Streamable HTTP`，URL 填 `http://localhost:8765/mcp`。
3. **確認容器在跑**：
   ```bash
   docker compose ps kc-mcp-server
   docker compose logs --tail=80 kc-mcp-server
   ```
4. **如果看到 `Not Acceptable`**：client 沒送 `Accept: application/json, text/event-stream`。

### 7.6 容器一直重啟

```bash
# 查看錯誤訊息
docker compose logs --tail=50 <container_name>

# 常見原因：
# - 密碼不一致 → 檢查 .env
# - 相依服務未啟動 → 檢查 depends_on 的服務
# - 設定檔語法錯誤 → 檢查 YAML/conf 檔案
```
